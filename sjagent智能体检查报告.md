## 二、问题明细
### 🚨 严重问题（必须修复，否则核心功能不可用）
#### 1. 知识库路径配置错误
- **问题**：config.yaml 中 `knowledge_base.path` 配置为 `"./src/knowledge/sjbzwiki"`，但实际知识库位于项目外的 `/e/sjagent/sjbzwiki` 目录
- **影响**：知识库功能完全失效，业务问答、规则查询等功能不可用
- **修复方案**：
  - 方案一：将 `sjbzwiki` 目录复制到 `/e/sjagent/sjagent/src/knowledge/` 下
  - 方案二：修改 config.yaml 路径为 `"../../sjbzwiki/sjbzwiki"`（相对路径）或 `"/e/sjagent/sjbzwiki/sjbzwiki"`（绝对路径）
#### 2. 缺少.env 环境变量文件
- **问题**：项目只有.env.example 示例文件，没有实际的.env 配置文件，缺少数据库、ERP、OSS、飞书、大模型等关键配置
- **影响**：系统无法连接任何外部服务，完全无法运行
- **修复方案**：复制.env.example 为.env，填写真实的配置信息
#### 3. 图片订单处理脚本为模拟实现
- **问题**：`scripts/image_order_detect.py` 核心功能均为模拟实现，没有实际的业务逻辑：
  - 黑框检测、图片裁切：模拟返回固定值
  - OCR 识别：模拟返回固定文本
  - OSS 上传：模拟生成 URL
  - 商品匹配：模拟返回固定商品 ID
  - 工作流订单创建：模拟返回结果
- **影响**：图片订单功能完全不可用
- **修复方案**：
  1. 使用 OpenCV 实现真实的黑色外框检测
  2. 使用 Pillow 实现图片裁切
  3. 集成 RapidOCR 实现真实的文字识别
  4. 集成 oss2 SDK 实现阿里云 OSS 上传
  5. 实现真实的商品匹配逻辑（查询 MySQL）
  6. 调用 ERP API 实现真实的工作流订单创建
#### 4. 件套换算逻辑未完整实现
- **问题**：需求要求用户报"X件"时，需从商品 `simple_desc` 字段提取换算率（如"规格:60套/件"），计算订单数量为 `X × 60` 套。当前代码未实现该逻辑
- **影响**：开单数量计算错误，导致库存扣减和订单金额错误
- **修复方案**：在订单处理节点中添加件套换算逻辑，从 `simple_desc` 中提取换算率，自动计算实际开单数量
#### 5. 商品查询不到时未触发用户确认
- **问题**：需求要求当商品查询不到（`product_warning`）时，必须询问用户确认商品名称，禁止自行猜测匹配。当前代码未实现该确认逻辑
- **影响**：商品匹配错误导致开单错误
- **修复方案**：在商品匹配节点中，当搜索不到匹配商品时，进入确认节点询问用户确认商品名称
#### 6. 本店出库未强制询问用户确认
- **问题**：需求要求只要发货仓库是自己店里（`warehouse_id=1`），无论什么流程都必须先询问用户确认。当前代码未实现该强制确认逻辑
- **影响**：可能未经用户同意就从门店发货，违反业务规则
- **修复方案**：在库存决策后，如果需要从门店发货，必须进入确认节点询问用户同意后才能继续
#### 7. 缺少 Flask 依赖
- **问题**：HTTP API 使用 Flask 框架，但 requirements.txt 中没有 Flask 依赖
- **影响**：HTTP API 渠道无法启动
- **修复方案**：在 requirements.txt 中添加 `flask==3.0.0`

### ⚠️ 中等问题（需要修复，影响部分功能）
#### 8. 烫金泡袋档位逻辑未实现
- **问题**：需求要求烫金泡袋类商品根据数量和单双面选择对应档位，当前代码未实现该逻辑
- **影响**：烫金泡袋类商品无法正确选择规格
- **修复方案**：在商品处理节点中实现烫金泡袋档位选择逻辑，根据配置规则自动匹配档位
#### 9. 打印规则未完整实现
- **问题**：需求要求齐唯茶叶（`customer_id=1`）开单后自动打印，其他客户默认不打印。当前代码未实现自动触发打印的逻辑
- **影响**：无法自动为齐唯茶叶客户打印订单
- **修复方案**：在开单成功后，判断客户 ID 是否为自动打印客户，如果是则自动调用打印 API 创建打印任务
#### 10. 统一开单逻辑需要验证
- **问题**：需求要求所有商品必须处理完所有库存操作（进货/调拨）后，再一次性开单，禁止分批开单。需要验证当前代码是否严格遵守该逻辑
- **影响**：如果分批开单会导致订单分散，增加财务和仓库管理成本
- **修复方案**：检查 `order_workflow_node` 和 `inventory_decision_node` 的实现，确保所有商品的库存操作完成后才调用一次 `sales_add`
#### 11. 库存决策逻辑需要验证
- **问题**：需求要求礼盒类商品需要查库存，非礼盒类直接待开单；百鑫无货时如果门店有货必须询问用户是否调货。需要验证当前实现是否符合要求
- **影响**：库存决策错误可能导致超卖、不必要的进货或调拨
- **修复方案**：对照 order-flow 中的库存决策表，验证 `inventory_decision_node` 的实现逻辑，确保完全符合规则
#### 12. 缺少飞书 SDK 依赖
- **问题**：飞书渠道需要飞书 SDK，但 requirements.txt 中没有相关依赖
- **影响**：飞书渠道无法启动
- **修复方案**：在 requirements.txt 中添加 `lark-oapi==1.2.0`（飞书官方 SDK）
#### 13. Dockerfile 需要完善
- **问题**：Dockerfile 比较简单，缺少必要的系统依赖（如 OpenCV 相关库），端口暴露不完整
- **影响**：Docker 部署失败
- **修复方案**：完善 Dockerfile，添加系统依赖，设置正确的工作目录，暴露需要的端口（5000/8080）

#### 14. 硬件设备对接入口为空
- **问题**：需求要求预留硬件设备对接入口（串口/网络设备），但 `src/channels/hardware/` 目录为空

- **优化建议**：在项目根目录创建 logs 目录，或在代码中自动创建日志目录
#### 16. 错误处理不够完善
- **问题**：部分节点的错误处理比较简单，可能导致异常信息直接暴露给用户
- **优化建议**：完善错误处理，对不同类型的错误返回友好的提示信息，记录详细的错误日志
#### 17. README.md 需要完善
- **问题**：README.md 内容比较简单，缺少部署说明、配置说明、使用示例等
- **优化建议**：完善 README.md，添加项目介绍、部署步骤、配置说明、使用示例、开发规范等
