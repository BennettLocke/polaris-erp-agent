版本：V1.0  
日期：2026-05-22  
适用项目：`sjagent`  
当前生产入口：`/web`  
目标新后台入口：未来新增 `/admin`  

> 本项目书用于指导后续 React + Radix 后台重构、服务层重构、数据库治理和业务规则统一。  
> 当前只写方案，不代表已经创建 React 源码。旧后台 `/web` 在新后台验收前必须继续可用。
>
> 后端服务层和路由现状以 [服务层与路由映射表](service_layer_route_map.md) 为准。React 新后台开发前必须先确认对应 API 已经走服务层。  
> React 页面调用接口以 [React 后台 API 合同](react_admin_api_contract.md) 为准。
> React 后台 UI 组件落地以 [React 后台 UI 组件匹配方案书](react_admin_ui_component_plan.md) 为准，不再按单个页面临时补组件。
> 2026-05-24 起，`/admin` UI 重构优先执行 [React Admin Radix/shadcn 底座重做项目计划书](react_admin_radix_shadcn_foundation_plan.md)，先搭完整组件体系，再逐页迁移。

## 1. 项目背景

现在北极星已经从“智能体辅助工具”逐步变成真正的业务系统：

- 商品已经迁移到自有商品库。
- 客户、用户、销售单、库存、出入库、盘点、调拨、图片资产、打印都已经开始走自有数据库。
- WebUI、Agent 对话、小程序、打印代理都在使用同一套后端接口。
- 旧 ERP 和 ShopXO 仍有部分历史参考代码，但目标是运行时不依赖旧 ERP/ShopXO。

当前后台已经能用，但架构开始吃力。主要问题不是界面单纯不好看，而是：

- 页面越来越多，原生 HTML + 一个超大的 `webui_api.js` 已经不适合继续堆功能。
- 关键业务逻辑分散在 API、前端 JS、Agent 工具、技能流程里。
- 钱、库存、月结、删除、回滚这些核心规则必须统一到后端服务层。
- 设置页需要变成结构化商业后台，而不是临时配置面板。
- 图片资产、商品、客户、销售单数据越来越多，需要分页、缩略图、懒加载和清晰分组。

所以这次重构的核心不是“换 React 皮肤”，而是：

```text
统一业务规则
  -> 建服务层
    -> 稳定 API
      -> 再做 React + Radix 后台
```

## 2. 项目目标

### 2.1 总目标

建设一个自有数据库、自有服务层、自有后台、自有小程序接口、自有 Agent 对话链路的业务系统。

### 2.2 具体目标

| 目标 | 说明 |
| --- | --- |
| 运行时不依赖旧 ERP | 旧 ERP 只作为迁移来源和参考，不作为日常业务运行依赖。 |
| 运行时不依赖 ShopXO 后台 | ShopXO 只作为历史数据来源或前端展示参考，核心后台由 sjagent 自己负责。 |
| 服务层统一 | WebUI、未来 React、Agent、打印、小程序都通过统一服务层访问核心业务。 |
| 小程序边界清楚 | 小程序不扣库存、不处理余额、不直接创建销售出库，只做展示、工作流、状态和查询。 |
| 前端组件化 | 新后台使用 React + TypeScript + Radix，避免继续维护超大原生 JS 文件。 |
| 数据结构稳定 | 商品、客户、用户、销售单、库存、图片资产、设置都以自有数据库为准。 |
| 操作可追踪 | 所有涉及钱、库存、订单状态的动作必须记录操作人、时间和变更日志。 |
| 可测试可验收 | 每个阶段都有执行标准和验收标准，不靠“看起来能用”。 |

### 4.2 后端目标栈

第一阶段不建议换后端框架。继续用 Flask，但补服务层。

| 层 | 选择 | 说明 |
| --- | --- | --- |
| Web 框架 | Flask | 保持当前可用性，降低重构风险。 |
| 服务层 | Python service classes | `SalesService`, `InventoryService`, `PaymentService` 等。 |
| 数据访问 | 继续使用 PyMySQL + `NativeDBClient`，逐步抽 Repository | 不立刻大换 ORM。 |
| 事务 | MySQL transaction | 所有钱/库存/订单状态变更必须在事务内。 |
| API | REST JSON | React、旧 WebUI、小程序、Agent 都调用统一 API。 |
| 权限 | `auth_user` + 角色/权限判断 | 先简单角色，后续细权限。 |
| 审计 | `audit_log` 或现有业务 log 表扩展 | 关键动作必须有操作人和前后值。 |

## 6. 后端服务层设计

### 6.1 服务清单

| 服务 | 职责 |
| --- | --- |
| `SalesService` | 创建销售单、删除销售单、销售单详情、销售单列表、打印前数据。 |
| `PaymentService` | 已付、微信、现金、余额、月结、未付、收款、结款。 |
| `CustomerBalanceService` | 充值、收款、调余额、余额明细、月结欠款计算。 |
| `InventoryService` | 扣库存、恢复库存、查库存、进货、盘点、调拨、库存日志。 |
| `ProductService` | SPU/SKU、编号、颜色、件规、1 件起、分类、单位。 |
| `MediaAssetService` | 上传、OSS、缩略图、绑定、待绑定、按 SPU 聚合。 |
| `WorkflowOrderService` | 工作流订单创建、状态、图片、删除、设计稿工作量。 |
| `SettingsService` | 编号、商品基础、库存规则、收款结款、图片/OSS、打印设置。 |
| `AuthService` | Web 登录、小程序 token、用户审批、角色和启用。 |
| `IdentityLinkService` | 统一处理手机号、微信 `openid/unionid`、后台账号 `auth_user`、客户 `party` 的绑定和解除绑定。 |
| `MiniAppService` | 小程序首页设计读取、商品货架聚合、用户中心订单统计。 |
| `PrintService` | 销售单模板、打印任务、打印预览。 |
| `AuditLogService` | 记录操作人、操作类型、变更前后数据、来源。 |

当前已落地的服务和路由映射详见 `docs/service_layer_route_map.md`。项目书里的服务清单是目标形态，实际开发时以映射表记录的当前状态为准。

### 6.2 服务层执行原则

| 原则 | 标准 |
| --- | --- |
| 前端不写核心业务规则 | React/旧 WebUI 只提交用户选择，不决定库存和余额怎么变。 |
| Agent 不直接改核心表 | Agent 调用 service 或 API，不自己写库存/余额 SQL。 |
| 小程序不扣库存 | 小程序工作流订单不影响 `inventory_balance` 和 `inventory_ledger`。 |
| 所有钱和库存变动必须事务化 | 销售单、余额、库存、日志必须在同一事务内。 |
| 所有关键操作必须有操作人 | Web、Agent、小程序都要识别当前用户。 |
| 身份绑定必须集中处理 | 后台用户改手机号、客户改手机号、微信手机号登录、小程序绑定手机号都必须走 `IdentityLinkService`，不能各页面各写一套匹配逻辑。 |
| 删除不等于物理删除 | 销售单、商品、客户默认软删除。 |
| 流水不可改历史 | 库存流水、余额流水只能新增反向记录，不直接改历史。 |

### 6.3 销售单创建标准流程

```text
创建销售单
  -> 校验当前用户权限
  -> 校验客户是否存在
  -> 判断客户是否月结
  -> 校验商品/SKU 是否存在
  -> 读取 SKU 库存规则
  -> 计算价格、数量、金额
  -> 创建 sales_order
  -> 创建 sales_order_item
  -> 对需要扣库存的 SKU 写 inventory_ledger
  -> 更新 inventory_balance
  -> 如果余额支付，写 customer_balance_ledger
  -> 写操作日志
  -> 返回销售单号
```

### 6.4 销售单删除标准流程

```text
删除销售单
  -> 校验权限
  -> 锁定 sales_order
  -> 判断是否已经删除
  -> 查询原销售出库 inventory_ledger
  -> 只恢复当初扣过库存的 SKU
  -> 不恢复不扣库存商品
  -> 如果余额支付，返还余额
  -> 如果已结月结单删除，转入余额或按规则处理
  -> 写库存反向流水
  -> 写余额流水
  -> 标记 sales_order.status = deleted
  -> 写删除人和删除时间
```

### 6.5 小程序边界

小程序明确不参与销售出库和扣库存。

| 小程序功能 | 是否允许 |
| --- | --- |
| 看商品 | 允许 |
| 看图片 | 允许 |
| 查库存 | 允许，只读 |
| 创建工作流订单 | 允许，不扣库存 |
| 更新制作/配送/丝印状态 | 允许 |
| 创建销售单并扣库存 | 不允许 |
| 删除销售单并回滚库存 | 不允许 |
| 收款/充值/结款/调余额 | 不允许 |
| 盘点/调拨/进货 | 第一阶段不建议给小程序开放 |

### 6.6 用户、客户和微信身份绑定标准

这部分用于解决后台账号、小程序微信账号和客户资料之间的关系，避免后续逻辑越来越绕。

| 对象 | 表 | 定位 |
| --- | --- | --- |
| 登录账号 | `auth_user` | 后台账号、小程序账号、内部员工和客户登录身份都在这里。 |
| 业务客户 | `party` | 客户资料，比如齐唯茶业、古越茗风；`kind='customer'`。 |
| 外部身份 | `auth_identity` | 微信 `openid/unionid`、小程序身份绑定，不直接决定权限。 |
| 账号绑定客户 | `auth_user.linked_party_id` | 登录账号关联到某个客户。 |

统一规则：

- 权限、启用、审批只看 `auth_user`。
- 客户资料、月结、电话、地址只看 `party`。
- 微信登录先查 `auth_identity`；没有绑定时再走手机号匹配。
- 后台用户设置手机号后，微信手机号登录应优先绑定到这个 `auth_user`。
- 客户后补手机号时，应自动查同手机号 `auth_user`，能唯一匹配时回填 `linked_party_id`。
- 内部账号 `admin/staff` 不能被客户手机号匹配逻辑强制改成客户角色。
- 一个手机号匹配多个账号或多个客户时，不自动绑定，进入待处理提示。
- 手动解除绑定只解除 `linked_party_id` 或 `auth_identity`，不删除账号、不删除客户。

标准流程：

```text
微信手机号登录 / 后台用户改手机号 / 客户改手机号
  -> 统一标准化手机号
  -> 查 auth_identity 是否已有微信绑定
  -> 查 auth_user.phone 是否唯一匹配
  -> 查 party.phone 是否唯一匹配
  -> 根据规则写 auth_identity 和 linked_party_id
  -> 记录绑定来源、时间和操作人
  -> 返回绑定结果或待人工处理原因
```

验收标准：

- 先小程序手机号登录，后客户补手机号，可以自动绑定客户。
- 先客户有手机号，后小程序手机号登录，可以自动绑定客户。
- 先后台内部账号设置手机号，后微信手机号登录，可以识别为该内部账号。
- 内部账号不会被客户绑定逻辑降级为客户。
- 重复手机号不会乱绑，必须提示人工处理。

当前落地状态：

- `IdentityLinkService` 基础服务已接入本地和服务器 `/opt/sjagent`。
- 服务器在 2026-05-24 已做过一次定向部署并重启 `sjagent.service`，验证 `/api/mini/home`、`/api/mini/goods/detail?id=12`、`/api/auth/wechat-quick-login`、`/web` 基础 smoke 通过。
- 目前后端可以接收小程序传来的真实手机号字段，比如 `phone`、`mobile`、`phoneNumber`、`purePhoneNumber`，并走统一绑定服务。
- 后续如果小程序只传微信 `phoneCode`，还需要补 `phoneCode -> 手机号` 的微信接口换号逻辑。这一步先记录，后面和小程序联调时再补。

- 写明影响表、字段、回滚方式。
- 不把测试脚本丢进主源码目录。

## 10. 执行标准

### 10.1 通用执行标准

每个任务必须满足：

- 先读现有代码和数据结构。
- 不破坏 `/web` 当前可用功能。
- 不把测试脚本、临时文件放进主源码目录。
- 不提交无关脏文件。
- 每个阶段有可运行验证。
- 每个业务动作有明确操作人。
- 涉及钱/库存必须有事务和日志。

### 10.2 后端执行标准

| 项目 | 标准 |
| --- | --- |
| API | 只做参数解析、权限校验、调用 service、返回 JSON。 |
| Service | 写业务规则、事务、调用 DB。 |
| DB Client | 只做 SQL 和数据映射，减少业务判断。 |
| 事务 | 销售、库存、余额、结款、删除必须事务。 |
| 错误 | 返回业务可读错误，不暴露原始英文异常。 |
| 日志 | 关键操作写业务日志和系统日志。 |
| 权限 | 后端强校验，不能只靠前端隐藏按钮。 |

### 10.3 前端执行标准

| 项目 | 标准 |
| --- | --- |
| 页面 | 每个主导航一个 feature。 |
| 组件 | 通用组件放 `components/ui`，业务组件放 `components/business`。 |
| API | 所有请求放 `api/*.ts`，页面不直接散写 URL。 |
| 状态 | 页面状态局部管理，跨页面状态再抽统一 store。 |
| 字段 | 不显示英文技术枚举。 |
| 表单 | 数字输入禁用滚轮误改。 |
| 图片 | 默认缩略图，预览才加载大图。 |
| 响应式 | 至少支持常用桌面宽度和基础移动宽度。 |

### 10.4 数据库执行标准

| 项目 | 标准 |
| --- | --- |
| 新表 | 必须有用途说明、字段说明、索引说明。 |
| 新字段 | 必须说明来源、默认值、是否可空。 |
| 迁移 | 必须可重复执行或有检查逻辑。 |
| 删除 | 业务主表优先软删除。 |
| 流水 | 库存和余额流水不可直接修改历史。 |
| 索引 | 查询页面使用字段必须有必要索引。 |

### 10.5 测试执行标准

测试不能只靠手点页面。

必须有：

- 服务层单元测试。
- API 集成测试。
- 数据库事务测试。
- 关键业务脚本测试。
- 前端构建测试。
- 关键页面浏览器验证。

测试数据放：

```text
tests/
scripts/
```

不能乱放到主源码目录。

| 工作流订单 | 是 | 是 | 是 | 是 | 是 | 创建/查看自己的 |
| 用户审批 | 是 | 否 | 否 | 否 | 否 | 否 |
| 系统设置 | 是 | 否 | 否 | 否 | 否 | 否 |

### 13.3 权限执行标准

- 前端可以隐藏按钮，但后端必须再次校验。
- 钱、库存、销售单删除、设置修改默认只有管理员。
- 如果要给员工开放删单或收款，必须在权限设置里显式开启。
- 小程序 token 必须只能访问小程序允许的接口。
- 小程序不能调用销售出库、调余额、盘点、调拨这些接口。

## 14. 备份、回滚和上线切换策略

### 14.1 数据备份标准

所有涉及数据库结构或批量数据修复的上线前必须做：

- 备份 `sjagent_core` 数据库。
- 记录备份文件名、时间、执行人。
- 备份当前代码版本 commit。
- 重要图片操作前确认 OSS 路径和本地上传目录。

### 14.2 数据库迁移回滚标准

每个迁移脚本必须写明：

- 影响哪些表。
- 新增哪些字段或索引。
- 是否可重复执行。
- 失败后如何回滚。
- 是否需要数据修复脚本。

对钱和库存相关迁移，必须先在测试库或复制数据上跑。

### 14.3 上线切换策略

新后台不能一次性替换旧后台。切换顺序：

1. `/web` 保持不变。
2. `/admin` 先只给管理员测试。
3. 先迁只读页面。
4. 再迁设置页。
5. 再迁客户和销售单。
6. 再迁商品和图片资产。
7. 再迁库存。
8. 每个模块验证通过后，旧页面按钮再逐步隐藏。
9. 最后 `/admin` 成为默认入口，`/web` 只保留一段时间作为回退。

### 14.4 回退标准

如果新后台出现严重问题：

- 立刻恢复使用 `/web`。
- 停止 `/admin` 对普通用户开放。
- 如果只是前端问题，不回滚数据库。
- 如果服务层导致数据错误，先冻结相关写操作，再按流水修复。
- 不直接删除历史流水，使用反向流水修正。

## 17. 业务规则冻结表

这些规则一旦确认，后续 WebUI、React 后台、Agent、小程序和服务器 API 都必须遵守。  
如果未来要改，只能通过设置项、迁移脚本和测试用例一起改，不能在某个页面或某个接口里单独改。

| 规则 | 标准 |
| --- | --- |
| 商品主编号 | 后续新建商品统一使用 `SJ` 编号，礼盒和泡袋都进入同一套编号池。 |
| 历史编号 | 历史乱编号只作为迁移参考，不作为新系统主键和业务判断依据。 |
| SKU 唯一依据 | 内部唯一编号是 `product_sku.sku_no`，也就是 `SJxxxx`。 |
| SPU 名称 | SPU 保存产品系列名称，比如“颜序半斤礼盒”。 |
| SKU 颜色 | SKU 保存具体颜色/规格，比如“红色”“黄色”“默认颜色”。 |
| SPU 可用颜色 | SPU 可以冗余保存可用颜色列表，用于创建商品时快速选择，不必每次反查 SKU。 |
| 默认颜色 | 没有明确颜色的 SKU 显示“默认颜色”，不能显示“未设置颜色”。 |
| 件规 | 件规显示为“件规：1件X套”，单位按商品单位展示。 |
| 1件起订 | 商品可设置是否 1 件起订，开单页和 Agent 都必须读取该规则。 |
| 泡袋库存 | 泡袋不扣库存。泡袋开单不查库存，删除销售单也不回滚泡袋库存。 |
| 礼盒库存 | 礼盒扣库存。开单成功扣库存，删除销售单回滚库存。 |
| 标签/服务类 | 不查库存、不扣库存、不回滚库存。 |
| 小程序库存 | 小程序不扣库存，不创建销售出库，不修改余额，只能做展示、查询和工作流。 |
| 普通客户默认结款 | 普通客户开单默认已付，默认已付方式为微信，除非手动选择其他方式。 |
| 月结客户默认结款 | 标记为月结客户后，开单页和 Agent 默认使用月结。 |
| 新客户默认规则 | 新建客户默认不是月结客户。 |
| 齐唯规则 | 只有明确设置为月结的月份/客户才按月结产生欠款，不能把所有单子都算成欠款。 |
| 余额规则 | 客户余额只来自销售单、收款、充值、结款、调余额等流水汇总，不能手工直接改最终余额。 |
| 余额扣减 | 使用余额付款时减少余额；删除该销售单时恢复对应余额。 |
| 月结欠款 | 月结单创建后形成负余额/应收；结款后改为已付并写入余额或收款流水。 |
| 销售单删除 | 第一阶段使用软删除，业务列表默认隐藏；删除时必须回滚库存和余额。 |
| 打印 | 销售单打印只打印销售单，不做进货单、采购单等模板。 |
| 操作人 | 开单、删单、结款、调余额、调库存、盘点、调拨、图片删除都必须记录操作人。 |
| 旧 ERP | 运行时不能依赖旧 ERP。迁移脚本可以读取旧 ERP 数据，但业务调用必须走 sjagent 自己的数据库和服务层。 |
| 测试数据 | 测试数据不能混入正式库，临时脚本不能散落在主源码目录。 |

## 18. 销售单状态机

销售单状态分两条线管理：  
一条是业务状态，一条是付款状态。不能把“已完成”和“已付款”混成一个字段。

### 18.1 业务状态

| 状态 | 含义 | 是否显示在默认列表 | 是否影响库存/余额 |
| --- | --- | --- | --- |
| `created` | 已创建，正常销售单 | 是 | 是 |
| `deleted` | 已删除/软删除 | 否 | 删除动作已回滚 |

第一阶段不做复杂的“已取消、已作废、已完成”一堆状态。  
业务上只需要“正常单”和“已删除单”，避免对账时看乱。

### 18.2 付款状态

| 状态 | 中文显示 | 含义 |
| --- | --- | --- |
| `paid` | 已付 | 现金、微信、余额、转账等已经结清。 |
| `monthly` | 月结 | 记入客户欠款，后续通过结款处理。 |
| `unpaid` | 未付 | 暂时未收款，也不属于月结规则。 |

### 18.3 已付方式

| 方式 | 中文显示 | 是否影响余额 |
| --- | --- | --- |
| `wechat` | 微信 | 不影响客户余额。 |
| `cash` | 现金 | 不影响客户余额。 |
| `balance` | 余额 | 扣减客户余额。 |
| `transfer` | 转账 | 不影响客户余额。 |

### 18.4 状态流转

| 场景 | 流转 |
| --- | --- |
| 普通客户开单 | `created + paid + wechat` |
| 月结客户开单 | `created + monthly` |
| 手动选择未付 | `created + unpaid` |
| 月结客户结款 | `monthly -> paid`，写收款/结款流水。 |
| 未付单收款 | `unpaid -> paid`，写收款流水。 |
| 删除已付微信/现金/转账单 | 软删除，礼盒库存回滚，付款状态保留在历史明细中。 |
| 删除余额付款单 | 软删除，礼盒库存回滚，客户余额恢复。 |
| 删除月结单 | 软删除，礼盒库存回滚，客户欠款减少。 |
| 删除不扣库存商品单 | 只处理金额和单据，不恢复库存。 |

## 20. API 合同清单

| --- | --- | --- | --- |

| `GET /api/sales/orders/{id}` | 销售单详情 | 销售单 ID | 单据、明细、付款、库存和操作日志 |
| `POST /api/customers/{id}/payments` | 收款 | 客户、金额、方式、备注、操作人 | 余额流水、客户余额 |
| `POST /api/customers/{id}/settlements` | 月结结款 | 客户、月份、金额、方式、操作人 | 被结销售单、合计、余额变化 |
| `POST /api/customers/{id}/balance-adjustments` | 调整余额 | 客户、金额、原因、操作人 | 余额流水、客户余额 |
| `GET /api/customers/{id}/balance-ledger` | 余额明细 | 客户、时间范围 | 明细列表、合计 |
| `POST /api/inventory/purchase` | 进货入库 | 商品、数量、仓库、操作人 | 入库单、库存流水 |
| `POST /api/inventory/stocktaking` | 盘点 | 商品、仓库、盘点数量、操作人 | 盘点单、差异流水 |
| `POST /api/inventory/transfer` | 调拨 | 商品、数量、出入仓库、操作人 | 调拨单、库存流水 |
| `POST /api/products` | 新建商品 | SPU、SKU、颜色、价格、件规、库存规则 | 商品 ID、SKU 编号 |
| `PATCH /api/products/{id}` | 编辑商品 | 商品字段、图片绑定、1件起订 | 保存结果 |
| `POST /api/media/assets` | 上传图片 | 图片、角色、绑定对象、操作人 | 图片资产 ID、缩略图地址 |
| `PATCH /api/media/assets/{id}/binding` | 绑定图片 | 图片 ID、SPU/SKU、角色 | 绑定结果 |
| `DELETE /api/media/assets/{id}` | 删除图片 | 图片 ID、操作人、原因 | 删除/解绑结果 |
| `GET /api/settings` | 设置读取 | 设置分组 | 设置数据 |
| `PATCH /api/settings` | 设置保存 | 设置分组、配置、操作人 | 保存结果和变更日志 |
| `POST /api/agent/tasks/sales-order` | Agent 开单 | 对话解析结果、当前用户、上下文 | 和销售单 API 一致的结果 |
| `POST /api/mini/workflow-orders` | 小程序工作流 | 客户、商品、数量、图片、状态 | 工作流单，不影响库存和余额 |

### 20.1 API 错误标准

| 场景 | 返回 |
| --- | --- |
| 商品不存在 | 明确提示哪个商品/颜色找不到。 |
| 礼盒库存不足 | 提示商品、仓库、当前库存、需要数量。 |
| 泡袋库存为空 | 允许开单，不报库存不足。 |
| 普通客户无付款方式 | 默认微信。 |
| 月结客户无付款方式 | 默认月结。 |
| 权限不足 | 提示当前账号不能执行该操作。 |
| 重复删除销售单 | 提示该单已删除，不重复回滚。 |
| 余额不足但选择余额付款 | 阻止开单，提示当前余额和差额。 |

## 21. Agent 对话链路验收标准

Agent 的验收不能只看“有没有回复”，要看是否调用正确服务、是否生成正确数据、是否速度可接受。

### 21.1 必测对话

| 场景 | 测试说法 | 预期 |
| --- | --- | --- |
| 普通开单 | “给古越茗风开半斤咖色 6 套” | 创建销售单，普通客户默认已付微信。 |
| 月结开单 | “给齐唯茶业开喜悦半斤红色 10 套” | 如果齐唯是月结客户，默认月结。 |
| 泡袋开单 | “给源袍开肉桂泡袋 50 个” | 不查库存、不扣库存。 |
| 礼盒库存不足 | “给某客户开喜悦半斤红色 9999 套” | 拦截并说明库存不足。 |
| 标签开单 | “给齐唯茶业开标签 2 张” | 允许开单，不查库存。 |
| 余额付款 | “用余额给客户开单” | 检查余额，不足则提示。 |
| 删除销售单 | “删除刚才那张销售单” | 软删除，回滚库存和余额，记录操作人。 |
| 查客户欠款 | “查齐唯 5 月欠款” | 返回 5 月月结单和合计。 |
| 查库存 | “查喜悦半斤红色库存” | 返回按仓库库存。 |
| 工作流 | “这个设计稿做 20 套，生成工作流单” | 只生成工作流，不扣库存。 |

### 21.2 性能标准

| 指标 | 标准 |
| --- | --- |
| 普通查询 | 2 秒内返回初步结果。 |
| 开单确认 | 先给确认摘要，再执行创建。 |
| 图片识别 | 可异步处理，先提示正在识别。 |
| 商品搜索 | 优先查本地商品索引和别名，不应每次大范围扫描。 |
| 客户搜索 | 优先按客户名、手机号、别名查本地数据库。 |

### 21.3 调用标准

- Agent 不能直接写数据库。
- Agent 不能调用旧 ERP 创建单据。
- Agent 只能调用服务层或服务层包装后的 API。
- Agent 每次写操作必须带当前登录用户或明确的操作来源。
- Agent 返回中必须显示关键业务结果：客户、商品、数量、金额、付款状态、库存影响。

## 22. 数据迁移和清洗标准

### 22.1 迁移范围

| 数据 | 标准 |
| --- | --- |
| 商品 | 以新 `product_spu`、`product_sku` 为准，历史编码只做参考。 |
| 图片 | 按 SPU 归集主图、详情页、颜色图，未识别的进入待绑定。 |
| 客户 | 修正“客户64、客户65”这类名称，必须绑定真实客户名。 |
| 销售单 | 默认历史单按已付处理，只有明确规则的客户/月才为月结。 |
| 余额 | 由销售单、收款、充值、结款、调余额流水重新汇总。 |
| 库存 | 以当前库存表和库存流水核对，不扣库存类商品不进入库存余额。 |
| 工作流单 | 保留状态和客户商品信息，不参与销售库存结算。 |

### 22.2 清洗规则

| 问题 | 处理方式 |
| --- | --- |
| 客户名是数字 ID | 查旧数据映射，改成真实客户名。 |
| 销售单全变欠款 | 按付款规则重算，普通客户恢复已付。 |
| 测试销售单 | 标记测试来源或删除，不能进入正式统计。 |
| 图片重复 | 同一文件 hash 去重，保留绑定关系。 |
| 主图按 SKU 重复 | 合并到 SPU 主图，只保留一张。 |
| 颜色图缺失 | 从 SKU 图片或详情页候选里补绑定。 |
| 商品编号空段 | 不回填历史，只保证后续从统一起点连续生成。 |

### 22.3 迁移验收

- 迁移前备份数据库。
- 迁移脚本必须能重复运行或明确禁止重复运行。
- 迁移后输出汇总：商品数、SKU 数、客户数、销售单数、图片数、库存流水数。
- 抽查重点客户：齐唯、古越茗风、藏茗聚仙。
- 抽查重点商品：喜悦、颜序、茶派、肉桂泡袋、大红袍泡袋。
- 抽查重点规则：泡袋不扣库存、小程序不扣库存、普通客户默认已付、月结客户默认月结。

## 26. 风险和控制

| 风险 | 控制方式 |
| --- | --- |
| React 重构影响当前业务 | `/web` 保留，新后台分阶段上线。 |
| 服务层重构引入金额/库存错误 | 先写测试，再切 API。 |
| 小程序误扣库存 | 后端禁止小程序调用销售出库服务。 |
| 图片资产卡顿 | 缩略图、分页、懒加载。 |
| 设置页配置错误 | 固定业务规则只读，修改写日志。 |
| 旧 ERP 逻辑残留 | 明确 runtime 禁止旧 ERP fallback。 |
| 测试数据污染业务库 | 测试脚本和测试库隔离。 |

## 27. 最终建议

不要先写 React 页面。正确顺序是：

1. 确认本项目书。
2. 冻结业务规则。
3. 做服务层。
4. 做服务层测试。
5. 再搭 React + Radix 底座。
6. 先迁移设置页。
7. 再迁移客户和销售单。
8. 再迁移商品和图片资产。
9. 再迁移库存。
10. 最后迁移 Agent 工作台。

这样做出来的新后台不是换皮，而是把北极星变成一个真正稳定、可追踪、可扩展的商业系统。
