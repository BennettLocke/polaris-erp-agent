# 小星语音助手架构重构项目书

## 1. 项目背景

当前小星语音助手同时承担本地唤醒、录音、ASR、TTS、屏幕显示、库存查询、商品纠错、客户资料处理、业务回复整理等职责。香橙派 Zero3 只有 2G 内存，适合做轻量设备端，不适合长期承载复杂业务逻辑。

本次重构目标是把香橙派从“业务处理机器”调整为“语音与屏幕终端”，把复杂数据处理放到服务器统一完成。

## 2. 核心目标

| 目标 | 说明 |
| --- | --- |
| 降低香橙派压力 | 香橙派只负责唤醒、录音、播放、屏幕展示 |
| 统一业务逻辑 | 库存、客户、商品、订单、纠错都由服务器处理 |
| 统一回答格式 | WebUI、语音助手、屏幕显示使用同一套结果 |
| 降低维护成本 | 后续新增礼盒、客户词、库存规则只改服务器 |
| 提高稳定性 | 香橙派少跑复杂逻辑，减少卡顿、误触、内存压力 |

## 3. 当前问题

| 问题 | 表现 | 根因 |
| --- | --- | --- |
| 设备端逻辑偏重 | 香橙派既监听又查数据又整理回复 | 设备端职责过多 |
| 回答不统一 | WebUI 和语音回复可能不一样 | 多处重复实现业务逻辑 |
| 热词维护麻烦 | 礼盒、客户、规格词要多处调整 | 词库没有集中生成 |
| 调试困难 | 需要同时看香橙派和服务器日志 | 边界不清晰 |
| 扩展风险高 | 屏幕、音乐、盘点、语音纠错叠加后更复杂 | 缺少统一设备 API |

## 4. 目标架构

```text
用户说话
  ↓
香橙派：本地唤醒
  ↓
香橙派：录音 / VAD / ASR
  ↓
服务器 API：意图识别 / 数据查询 / 结果整理
  ↓
服务器返回：
  - speak：语音播报文本
  - display：屏幕展示数据
  - action：设备下一步动作
  ↓
香橙派：播放语音 / 更新屏幕
```

## 5. 职责划分

| 模块 | 所在位置 | 职责 |
| --- | --- | --- |
| 本地唤醒 | 香橙派 | 只判断“小星小星”是否唤醒 |
| 录音与 VAD | 香橙派 | 判断开始说话和结束说话 |
| ASR | 短期香橙派，后续可服务端代理 | 把音频转成文字 |
| TTS 播放 | 香橙派 | 播放“在呢”和业务结果 |
| 屏幕页面 | 香橙派 | 展示待机页、业务结果、按钮事件 |
| 意图识别 | 服务器 | 判断库存、客户、开单、聊天等意图 |
| 商品/客户纠错 | 服务器 | 处理“喜越→喜悦”“见喜”“半斤”等业务词 |
| 数据查询 | 服务器 | 查 MySQL、ERP、自有业务库 |
| 回复整理 | 服务器 | 生成短语音回复和详细屏幕数据 |
| 日志监控 | 服务器 + 香橙派 | 记录 ASR、API、查询、TTS 耗时 |

## 6. 核心 API 设计

### 6.1 语音命令 API

`POST /api/device/voice/command`

香橙派把 ASR 识别后的文字发送给服务器，服务器返回语音播报和屏幕展示数据。

请求示例：

```json
{
  "device_id": "orangepi-xiaoxing-01",
  "session_id": "20260531-001",
  "trace_id": "voice-20260531-001",
  "source": "voice",
  "text": "喜悦半斤红色库存",
  "asr_confidence": 0.86
}
```

返回示例：

```json
{
  "ok": true,
  "trace_id": "voice-20260531-001",
  "intent": "inventory_query",
  "speak": "喜悦半斤，百鑫红色6套，黄色14套，橙色1套。",
  "display": {
    "mode": "inventory_result",
    "title": "喜悦半斤库存",
    "summary": "百鑫21套，自己店里8套",
    "items": [
      {"warehouse": "百鑫仓库", "color": "红色", "qty": 6},
      {"warehouse": "百鑫仓库", "color": "黄色", "qty": 14},
      {"warehouse": "百鑫仓库", "color": "橙色", "qty": 1},
      {"warehouse": "自己店里", "color": "红色", "qty": 8}
    ]
  },
  "device_action": {
    "next_state": "idle",
    "listen_again": false,
    "command_window_seconds": 2,
    "screen_mode": "result"
  },
  "timing": {
    "server_ms": 280,
    "query_ms": 120,
    "reply_ms": 35
  }
}
```

### 6.2 屏幕状态 API

`GET /api/device/screen/state?device_id=orangepi-xiaoxing-01`

用于香橙派屏幕定时拉取当前状态，例如待机、正在听、正在处理、库存结果。

### 6.3 屏幕事件 API

`POST /api/device/screen/event`

用于上报屏幕点击事件，例如点击某个库存数字进入盘点。

请求示例：

```json
{
  "device_id": "orangepi-xiaoxing-01",
  "event": "inventory_qty_click",
  "payload": {
    "product_id": 123,
    "warehouse": "百鑫仓库",
    "color": "红色",
    "current_qty": 6
  }
}
```

### 6.4 设备心跳 API

`POST /api/device/heartbeat`

用于记录香橙派是否在线、当前版本、内存、服务状态和最近错误。

### 6.5 ASR 热词 API

`GET /api/device/asr/hotwords?device_id=orangepi-xiaoxing-01`

用于香橙派开机或定时拉取火山 ASR 热词。热词由服务器统一生成，避免香橙派本地维护多份礼盒名、客户名和规格词。

返回示例：

```json
{
  "version": "20260531-001",
  "ttl_seconds": 3600,
  "hotwords": [
    "小星",
    "库存",
    "客户",
    "礼盒",
    "见喜",
    "喜悦",
    "半斤",
    "三两",
    "百鑫仓库",
    "自己店里"
  ]
}
```

### 6.6 屏幕推送 API

`GET /api/device/screen/events?device_id=orangepi-xiaoxing-01`

优先使用 SSE `EventSource` 推送屏幕状态。服务器有新状态时主动推给屏幕页面，减少轮询和画面卡顿。

事件示例：

```json
{
  "trace_id": "voice-20260531-001",
  "mode": "inventory_result",
  "title": "喜悦半斤库存",
  "summary": "百鑫21套，自己店里8套",
  "items": [
    {"warehouse": "百鑫仓库", "color": "红色", "qty": 6}
  ]
}
```

## 7. 回答格式规则

| 类型 | 语音播报 `speak` | 屏幕展示 `display` |
| --- | --- | --- |
| 库存查询 | 简短，直接说产品和数量 | 显示仓库、颜色、数量、合计 |
| 查客户 | 只说关键欠款、最近订单、联系方式 | 显示客户卡片和明细 |
| 开单/调货 | 只说确认信息和下一步 | 显示订单草稿 |
| 没听清 | “没听清，再说一遍。” | 显示重试提示 |
| 无结果 | “没找到这个商品，你再说一下名称。” | 显示候选商品或空状态 |
| 服务器异常 | “服务器没连上，稍后再试。” | 显示离线状态 |

## 8. 设备会话状态机

香橙派不再靠零散的计时器自行判断下一步，而是维护明确状态。服务器返回 `device_action.next_state` 后，设备端按状态切换。

| 状态 | 含义 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| `idle` | 待机 | 开机、播报结束、超时结束 | 本地唤醒命中 |
| `wake` | 已唤醒 | 命中“小星小星” | 唤醒提示音播放完成 |
| `listening` | 等待用户指令 | 唤醒提示音结束 | 用户说完、等待超时 |
| `processing` | 服务器处理中 | ASR 文本已发给服务器 | 收到服务器结果或失败 |
| `speaking` | 正在播报 | 需要播放 `speak` 或本地提示音 | 播放完成 |
| `cooldown` | 冷却 | 播放完成后短暂忽略自身声音 | 冷却时间结束 |

状态规则：

1. 唤醒回应播放期间不采集命令，避免把“在呢”识别成用户指令。
2. `listening` 默认等待 2 秒，无有效语音则回到 `idle`。
3. 用户说“没事了”“不用了”“取消”时直接进入 `idle`。
4. ASR 结果过短或不确定时，不进入业务查询，直接返回“没听清，再说一遍”。
5. 服务器可以按不同任务返回不同 `command_window_seconds`，但设备端默认不超过 5 秒。

## 9. TTS 分层策略

TTS 不采用单一方案，按使用频率和实时性分层处理。

| 类型 | 处理方式 | 原因 |
| --- | --- | --- |
| 唤醒回应 | 香橙派本地缓存 WAV | “在呢 / 我在 / 小星在”必须最快播放 |
| 短错误提示 | 香橙派本地缓存 WAV | “没听清”“服务器没连上”不需要实时合成 |
| 处理中提示 | 超过 1 秒再播放 | 0 秒结果不打断业务播报 |
| 业务查询结果 | 服务器生成文本，香橙派播放或按需 TTS | 保证 WebUI、语音、屏幕结果统一 |
| 长回复 | 服务器先压缩成短播报 | 避免语音废话太多 |

唤醒回应轮播顺序：

```text
我在 → 在呢 → 小星在
```

处理中提示规则：

1. 服务器 1 秒内返回结果时，不播放“正在处理”。
2. 超过 1 秒未返回时，播放一次短提示。
3. 处理中提示不进入屏幕主消息流，避免污染业务结果。

## 10. 热词生成与刷新策略

火山 ASR 热词由服务器统一生成，香橙派只拉取使用。热词不能完全依赖 active 商品，因为下架、删除、分类异常会导致常用词消失。

热词分三层：

| 层级 | 示例 | 来源 |
| --- | --- | --- |
| 固定必带词 | 小星、库存、客户、礼盒、半斤、三两、见喜、喜悦 | 配置文件 |
| 业务常用词 | 百鑫仓库、自己店里、红色、黄色、咖色、长半斤、短半斤 | 服务器业务词库 |
| 数据库动态词 | 客户名、在售礼盒名、常用商品系列 | MySQL / ERP |

刷新规则：

1. 香橙派开机后请求 `/api/device/asr/hotwords`。
2. 每小时刷新一次，服务器返回 `ttl_seconds`。
3. 服务器热词响应带 `version`，版本不变时香橙派不用重载。
4. 固定必带词不受商品状态影响，`见喜`、`喜悦`这类常用系列必须一直保留。
5. 热词变更由服务器日志记录，方便排查“为什么某个词没识别到”。

## 11. 屏幕推送机制

屏幕页面优先使用服务器推送，不建议高频轮询。

推荐方式：

```text
香橙派浏览器打开屏幕页面
  ↓
页面建立 SSE / EventSource 连接
  ↓
服务器推送待机、正在听、查询结果、错误状态
  ↓
页面局部更新，不整页刷新
```

降级方式：

1. SSE 不可用时，每 2 秒请求一次 `/api/device/screen/state`。
2. 查询处理中可以临时缩短到 500ms。
3. 查询完成后恢复低频刷新。

屏幕显示规则：

1. `processing` 不显示成主要对话内容，只显示轻量状态。
2. 查询结果优先显示 `display`，不直接显示大模型原文。
3. 库存结果过长时扩大结果区或分页，不遮挡机器人。
4. 点击库存数字时，上报 `inventory_qty_click`，带原始库存数量，进入盘点流程。

## 12. trace_id 与耗时日志规范

每次语音交互必须有一条贯穿全链路的 `trace_id`。

链路包含：

```text
wake_detected
asr_start
asr_done
server_request
intent_resolved
data_query
reply_built
tts_start
tts_done
screen_updated
```

日志示例：

```json
{
  "trace_id": "voice-20260531-001",
  "device_id": "orangepi-xiaoxing-01",
  "asr_text": "喜悦半斤红色库存",
  "intent": "inventory_query",
  "asr_ms": 1320,
  "api_ms": 80,
  "query_ms": 210,
  "reply_ms": 50,
  "tts_ms": 620,
  "total_ms": 2280
}
```

排查原则：

1. 用户说“慢”，先看 `asr_ms`、`query_ms`、`tts_ms` 哪个最高。
2. 用户说“没反应”，先查同一 `trace_id` 是否进入 `asr_done`。
3. 用户说“查错”，先查 `asr_text` 和服务器纠错结果。
4. 用户说“屏幕没变”，先查 `screen_updated` 是否发送。

## 13. 没听清与纠错策略

服务器必须先判断 ASR 文本是否足够可靠，再决定是否查询业务数据。

直接返回“没听清”的情况：

| ASR 文本 | 原因 |
| --- | --- |
| 嗯 | 信息不足 |
| 那个 | 信息不足 |
| 查一下 | 缺少商品或客户 |
| 半斤 | 只有规格，没有系列 |
| 库存 | 只有意图，没有对象 |
| 裤子库存 | 与业务词不匹配，疑似误识别 |

允许纠错后继续查询的情况：

| ASR 文本 | 纠错结果 |
| --- | --- |
| 喜越半斤库存 | 喜悦半斤库存 |
| 喜语半斤库存 | 喜悦半斤库存，必要时返回候选 |
| 小心小心 | 小星小星，仅用于唤醒等价词 |
| 见喜三两裤存 | 见喜三两库存 |

纠错规则：

1. 只对礼盒系列、规格、颜色、仓库、客户名做业务纠错。
2. 纠错置信度不足时返回候选，不直接执行。
3. 语音播报只说最终业务结果，不解释复杂纠错过程。
4. 屏幕可以展示“按喜悦半斤查询”，方便确认。

## 14. 香橙派 Thin Client 边界

重构后的香橙派只保留设备能力，不保留复杂业务能力。

建议保留文件/模块：

```text
scripts/wake_listener.py
src/device/device_client.py
src/device/audio_player.py
src/device/screen_client.py
data/generated/voice/prompts/
models/openwakeword/
```

不建议保留在香橙派执行的能力：

1. 直接查 MySQL。
2. 直接查 ERP。
3. 复杂商品匹配。
4. 客户资料整理。
5. 库存回复格式生成。
6. 大模型多轮业务判断。

香橙派只做三件事：

1. 听到唤醒。
2. 把用户命令传给服务器。
3. 播放和展示服务器返回结果。

## 15. 分阶段实施计划

| 阶段 | 目标 | 工作内容 | 验收标准 |
| --- | --- | --- | --- |
| 第 1 阶段 | 明确设备边界 | 香橙派只保留唤醒、ASR、TTS、屏幕 | 香橙派不直接查库存、不整理业务结果 |
| 第 2 阶段 | 新增服务器命令 API | 实现 `/api/device/voice/command` | 终端传文字能返回 `speak`、`display`、`device_action` |
| 第 3 阶段 | 增加状态机 | 香橙派按 `idle/listening/processing/speaking` 切换 | 不再出现没说完就执行、播报声被识别的问题 |
| 第 4 阶段 | 迁移库存回复 | 库存查询统一由服务器生成语音短回复 | “喜悦半斤库存”返回固定格式 |
| 第 5 阶段 | 香橙派接入 API | ASR 文本发服务器，播放服务器 `speak` | 语音查询能完整走服务器 |
| 第 6 阶段 | 屏幕数据接入 | 屏幕显示服务器 `display` | 查询结果显示在待机屏幕左侧 |
| 第 7 阶段 | 屏幕推送 | 优先 SSE 推送，轮询作为降级 | 查询结果能主动刷新到屏幕 |
| 第 8 阶段 | 词库集中 | 商品名、客户名、规格词统一生成 | 火山热词、纠错词、查询词来源一致 |
| 第 9 阶段 | 降级处理 | 服务器断开时给出明确提示 | 断网不死等、不误查 |
| 第 10 阶段 | 性能监控 | 记录唤醒、ASR、API、查询、TTS 耗时 | 日志能定位慢在哪个环节 |

## 16. 性能预估

| 环节 | 预计耗时 |
| --- | ---: |
| 本地唤醒 | 0.2-1.0 秒 |
| 录音结束判断 | 0.5-1.0 秒 |
| ASR | 0.8-2.0 秒 |
| 请求服务器 API | 0.05-0.3 秒 |
| 服务器查询库存 | 0.1-0.8 秒 |
| 服务器整理回复 | 0.1-1.0 秒 |
| TTS 开始播放 | 0.3-1.0 秒 |

业务处理迁到服务器通常不会明显增加延迟。主要慢点仍然是 ASR、录音结束判断和 TTS。服务器处理数据比香橙派更快，也更稳定。

## 17. 关键实现原则

1. 服务器是唯一业务大脑。
2. 香橙派不直接查数据库。
3. 语音回复和屏幕展示分开生成。
4. 语音回复必须短，屏幕展示可以详细。
5. 不确定时优先返回“没听清”或“没找到候选”，不胡乱查询。
6. 所有设备请求都带 `device_id` 和 `session_id`。
7. 所有关键耗时写日志，方便后续优化。
8. WebUI、语音、屏幕共用同一套业务服务。
9. 唤醒回应、短错误提示优先本地缓存，业务结果由服务器统一生成。
10. 热词由服务器统一生成，固定必带词不受商品状态影响。

| 礼盒热词 | 见喜半斤库存 | 能识别见喜并查询 |
| 没听清 | 只有一个字或无效词 | 回复“没听清，再说一遍” |
| 服务器断开 | 停止服务器后查询 | 香橙派提示服务器没连上 |
| 屏幕显示 | 查询库存后 | 左侧展示详细库存，不挡机器人 |
| 盘点入口 | 点击库存数字 | 进入对应商品、颜色、仓库的盘点状态 |
| 状态机 | 唤醒后沉默超过 2 秒 | 自动回到待机 |
| 取消命令 | 没事了 | 不继续追问，直接待机 |
| 热词刷新 | 请求 `/api/device/asr/hotwords` | 返回固定词、业务词、动态词和版本号 |
| 链路日志 | 完成一次库存查询 | 同一 `trace_id` 串起 ASR、API、查询、TTS、屏幕 |

## 19. 最终验收标准

1. 香橙派只负责设备能力，不再直接处理复杂业务。
2. 服务器 API 可以独立完成库存查询和回复整理。
3. WebUI、语音助手、屏幕展示的数据来源一致。
4. 查询“喜悦半斤库存”“见喜半斤库存”等业务词能稳定返回。
5. 普通库存查询从说完到开始回答控制在 1.5-3 秒左右。
6. 服务器或网络异常时有明确降级提示。
7. 后续新增礼盒、客户、规格词只需要改服务器侧词库和业务逻辑。
8. 每次语音交互都有 `trace_id`，能定位慢点和错点。
9. 屏幕结果由服务器推送或低频拉取，不靠高频刷新硬撑。
10. 香橙派端可以被看作 Thin Client，不再需要同步完整业务规则。

## 20. 推荐执行顺序

优先执行第 2、3、4、5 阶段：

1. 先在服务器实现 `/api/device/voice/command`。
2. 同时返回 `speak`、`display`、`device_action` 和 `trace_id`。
3. 增加香橙派状态机，让设备按服务器返回动作切换。
4. 把库存查询结果整理为 `speak` 和 `display`。
5. 香橙派唤醒后只把 ASR 文本发给服务器。
6. 香橙派播放服务器返回的 `speak`。
7. 再把 `display` 接入屏幕页面。

这样可以最快验证新架构是否有效，也不会一次性改太多设备端代码。
